## Objective

Fix the second root issue: when retrieval yields a PDF, the MCP server should extract readable text so the LLM can actually consume the paper instead of storing a placeholder only.

## Current Problem

- `parsePaper(..., "pdf")` currently returns a placeholder record only.
- This means:
  - retrieval may be marked as success
  - `_parsed` may exist
  - but `extracted_text` is just a note and not the paper content
- As a result, the LLM has not actually read the PDF.

## External References

- Mozilla PDF.js / `pdfjs-dist`: general-purpose PDF parsing in JavaScript
- `unpdf`: JavaScript PDF extraction wrapper built for Node/serverless and AI-style text extraction
- These are candidate implementation references if no built-in local parser exists

Chosen reference:

- `unpdf` because it wraps PDF.js cleanly for Node/serverless use and fits the MCP server architecture without requiring Python or system binaries

## Constraints

- No Python interpreter is available in the current shell environment
- No `pdftotext` binary was found
- Prefer a local Node-based solution integrated into the MCP server

## Desired Outcome

- PDF retrieval should produce real text extraction whenever the PDF contains selectable text
- If extraction fails, the record should clearly say why
- Keep the architecture local-first and batch-friendly

- 对照用户提供的站点列表与截图，判断问题发生在哪一层

## Current Findings

1. 运行时默认启用 `browser_assisted=true`、`elsevier_api=true`、`springer_oa_api=true`，因此结构化路由失败后会自动回退到浏览器路由。
2. 项目根目录与 `mcp/paper-search-agent-mcp` 目录下都存在 `.env`；服务入口 `src/server.ts` 使用了 `import "dotenv/config"`，因此 MCP 进程正常启动时会读取 `.env`。
3. 本地 shell 里最初看到 `ELSEVIER_API_KEY=NOT_SET` 只是因为普通 PowerShell 会话没有自动 source `.env`，不能代表 MCP 服务器进程的实际环境。
4. 历史审计日志显示 Elsevier API 曾成功获取多个 DOI 的 `fulltext.xml`，例如 `10.1016/j.cell.2020.06.009`、`10.1016/j.neuron.2023.07.002`、`10.1016/j.jmp.2024.102843`，说明项目并非总是绕过 Elsevier API。
5. 在实际运行目录 `C:\Users\STSna\.codex\mcp\paper-search-agent-mcp` 中，`.env` 存在且 `ELSEVIER_API_KEY` 已配置，审计日志也证明该实例确实使用过 `elsevier_api_fulltext` 并成功。
6. 与用户这次最接近的一批运行日志显示：
   - `10.1016/j.compstruct.2009.03.001` -> `elsevier_api_fulltext` 成功
   - `10.1016/j.ultras.2012.08.006` -> `elsevier_api_fulltext` 成功
   - `10.1016/j.jsv.2008.08.030` -> `elsevier_api_fulltext` 成功
   - `10.1088/0256-307x/18/12/324` -> `browser_download_pdf` 返回 `NEEDS_HUMAN`，随后自动落到 `manual_import_pdf`
   - `10.1007/s11467-006-0016-x` -> `browser_download_pdf` 成功抓取 HTML
   - `10.1177/1077546315619264` -> `browser_download_pdf` 在 `https://doi.org/...` 导航时报 `ERR_CONNECTION_CLOSED`
7. `access-planner.ts` 会把 `browser_download_pdf` 无条件追加到候选路由末尾，只要浏览器辅助开启。
8. `fulltext-fetcher.ts` 的浏览器路由固定从 DOI 落地页 `https://doi.org/<doi>` 开始，不做出版社特化处理。
9. `playwright-retriever.ts` 每次导航都会 `newPage()`，不会复用或关闭前一个页面，因此批量处理时会逐渐堆积标签页。
10. `tryDownloadPdf()` 只监听 `download` 事件和“当前页直接跳到 PDF”这两种情况，没有处理 `target=_blank`/popup 新标签页。
11. ScienceDirect 常见的 `pdf?md5=...` / `pdfft?md5=...` 是会话型、临时型 PDF 地址；当前实现若直接跳到这些地址，容易得到错误页而不是稳定下载。
12. `browser_retrieve` 适配器本身支持 `navigate -> 用户处理验证 -> check` 的双阶段流程，但 `fetch_fulltext` 里的 `executeBrowserRoute()` 只调用了 `navigate`，没有任何地方在自动流程里继续调用 `check`。

## Likely Root Cause

- 前提修正：不是因为 `.env` 没配置，也不是因为项目完全没有用 Elsevier API。
- 更可能的事实链是：该实例里 Elsevier API 本身可用，但用户截图中的那两个 ScienceDirect 标签不一定来自主检索链路里的 `elsevier_api_fulltext` 失败，也可能来自后续单独的 OA/浏览器测试或浏览器点击 PDF 的派生页面。真正可由日志确认的故障点主要集中在非 Elsevier 站点的浏览器路线，以及浏览器适配器自身的状态机缺陷。
- 主问题一：浏览器适配器没有正确处理点击 PDF 后新开标签页的场景，导致多个标签页被不断打开但不受控。
- 主问题二：自动检索链路没有把浏览器路线的人工交互闭环接起来，遇到验证后不会停在当前论文上等待 `check`，而是把这次尝试当失败继续后续流程。
- 主问题三：Elsevier/ScienceDirect 的 PDF 获取逻辑过于通用，错误地把临时 PDF 链接当成稳定下载地址处理，最终落到错误页面。
- 次问题：`executeBrowserRoute()` 把浏览器状态目录写死为 `./browser-state`，没有使用配置里的 `config.browser.state_directory`，虽然这不是截图中的直接故障点，但会让状态复用更不稳定。
